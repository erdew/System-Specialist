Print Server

Bir şirketin Printer ihtiyacı nasıl karşılanır?

Printer nedir?

Çıktı alan cihaz Printer Device'tır. Control Panel'de Device'lar bölümünde görüntülenen Device ise aslında Printer'dır. Printer da aslında bir Driver'dır. Printer Device ve Printer kavramları arasında bir fark yokmuş gibi görünür fakat ciddi farklar vardır. Bunu örneklerle açıklayacağız.

Network'lere bağlanabilen Printer Device'lar da vardır. Bunlara Network Printer Device adıverilir. Bu Printer Device'ların üzerinde CPU, RAM vs. gibi donanım birimleri bulunur. Network Printer Device'larını Printer Device'lardan ayıran özellik üzerlerinde Ethernet Port'unun olmasıdır. Doğal olarak bu Ethernet Port'una IP atanabilir ve bu IP sayesinde Network ile iletişim kurabilirler. Network Printer Device'lar normal Printer Device'lara göre daha maliyetlilerdir. Şirketler genel olarak orta maliyetli, ortalama Printer Device'lar kullanırlar. Network Printer'lar çok professiyonel ortamlarda kullanılırlar. Çok sayıda kullanıcı ve departman, çok sayıda çıktı almak durumundaysa Network Printer kullanmak daha mantıklı ve avantajlı olur.

Printer oluşturmak ve Printer Device'lara paylaşım gönderebilmek için Print Server kurulması gerekir. Print Server, bir Server'a kurulabileceği gibi Client Windows'lara da kurulabilir. Printer Device'ları, oluşturduğumuz Print Server'a bağlayarak; İzinler, Kuyruk Sıralaması vs. gibi ayarları düzenleyip, Printer Device'ların yaptığımız ayarlara göre çıktı almasını sağlayabiliriz. Daha professiyonel bir tasarım yapılacak ise Print Server Client'e değil Server'a kurulmalıdır. Üzerine Print Server kurulan makine, Printer Server olur. O isimle anılır. Normalde Domain Controller kesinlikle Print Server yapılmaz fakat eğitimde yapmak bir sorun oluşturmaz. Ancak yine de uygularken yeni bir makine üretip onu Print Server yapmak en doğrusudur. Print Server olarak belirlenecek makinenin donanımı performanslı olmalıdır.

Print Server aslında bir derleyicidir. Çıktı için yapılan paylaşımları Printer Device'ın anlayacağı dile derler.
Server Manager > Tools > Services > Print Spooler
Bir makinenin Print Server olabilmesi için Print Spooling servisi açık olmalıdır. Kapalı ise bu sunucuya bağlı olan Printer Device'a paylaşım yapılsa bile o Printer Device paylaşımı algılayamaz doğal olarak çıktı alınamaz. Çünkü o paylaşım Printer Device'ın anlayabileceği bir dile çevrilememiştir.
Printer Device'ın üzerindeki iş yükünden dolayı vs. bir şekilde donduğunu varsayalım. Bu sorunu çözmek için aşağıdaki yol izlenebilir;
Print Spooler > Right Click > Restart yapılır.
Restart yapıldığı zaman bütün kuyruk silinir. O anda yapılan işlem devam eder.

Print Device'a bağlanmak için;
Çok eski Printer Device'larda LPT Connector kullanılır. LPT Connector'e aynı zamanda "Paralel Port" adı da verilmiştir.
Günümüzde Printer Device'a bağlanmak için en çok USB Connector kullanılır.
Ethernet Port'u veya WiFi da kullanılabilir.

Printer Device ile Print Server'ın fiziksel bağlantısı yapıldıktan sonra Print Server'ın kurulum aşaması

Server Manager > Print Management > Print Servers > PC(Local, Print Server olarak belirlediğimiz makine) > Printers
Bu konumda gördüğümüz Printer'lar içinde Sanal Printer'lar da vardır. Örneğin Microsoft Print to PDF. Bu Printer ile örneğin bir web sitesinin PDF formatında kaydedilmesini sağlayabiliriz.
Yukarıda belirttiğimiz Printers konumunda Right Click > |+|Add Printer > Search the Network for printers || |+|Add a TCP/IP or Web Services Printer by IP Address or Hostname || |+|Add a new Printer using an existing port || |+|Create a new port and add a new printer.
Duruma göre yukarıdaki seçeneklerden birisi seçilir. Add a new printer using an existing port'u seçersek, fiziksel olarak hangi Port'tan bağladığımızı seçeriz. Install ederek devam ederiz, bize bir marka girmemizi sorar, fiziksel olarak bağlayacağımız Printer Device'ın marka modelini gireriz. Daha sonra isim verilir. |+|Share this Printer seçilirse o Printer Network'teki diğer cihazlar tarafından görüntülenebilir, bağlanılabilir durumda olur. Az önce verdiğimiz isim Share Name'dir Printer'ın hangi isimle görüntüleneceğini belirler. Daha sonra bu Printer Device'ın binada hangi konumda(Location) olduğu belirtilir. Ardından Next ve Finish seçilerek Printer'ın kurulumunu tamamlamış oluruz.
En üstte belirttiğimiz konumda bu Printer listelenmiş olur.

Network'te bulunan bir son kullanıcı Print Server'a Network üzerinden bağlantı kurar; örneğin \\pc1\ -> Bu konumda bir önceki paragrafta kurulumunu yaptığımız Printer'ı görüntüleyebilir. Çünkü o Printer'ı Share etmeyi seçmiştik. Kullanıcı bu Printer'a sağ tıklayıp Connect yapabilir. Windows 7, Windows 10 gibi Windows'un çeşitli versiyonlarında Printer'ları çalıştıracak olan Driver'lar farklıdır. Bu sebeple ağdaki her tipten işletim sistemi için gerekli Driver'lar Print Server'ın Driver'ları ile uyumlu olmalı. Driver'lar uyumlu olmalı ki Connect yapılabilsin. Driver'ları yüklemek için şu konuma gidilir;
Server Manager > Print Management > Print Servers > PC(Local, Print Server olarak belirlediğimiz makine) > Printers > Kurduğumuz Printer > Right Click > Properties > Sharing > Additional Drivers -> Bu konumda, örneğin çok eski bir Windows'tan Connect olunmak istenirse onun Driver'ı indirilip, buradan yüklenir.
Printer'lar Network'e varsayılanda sunulmazlar, bu sebeple Network'teki kullanıcılar Printer Device eklemek istediğinde bu Printer'ları göremezler. Bunu açmak için; yukarıdaki konumdan List in the Directory ile tamamlandıktan sonra bu sorun ortadan kalkar.

Yukarıda bahsettiğimiz Directory, Active Directory'dir. Server, Printer oluştururken ya da oluşturulduktan sonra Sharing Tab'ında Share Name'inin sonuna $ işareti eklersek Printer, Hidden olarak paylaşılır.

Permissions

Server Manager > Print Management > Print Servers > PC(Local, Print Server olarak belirlediğimiz makine) > Printers > Kurduğumuz Printer > Right Click > Properties > Security > Permissions >
|+|Print -> ||
|+|Manage Documents -> Printer'daki iş kuyruğuna müdahale edebilir, kuyruktaki işleri kaydırabilir, duraklatabilir vs.
|+|Manage This Printer -> Bu Printer'ın Driver'larını ve Printer'ın kendisini değiştirebilir, kurabilir ve kaldırabiliriz.
Print Operators'ın Full Permission hakları vardır.
Örneğin bir Printer'ımız var bu Printer'ı birden varsa Group yani departman kullanıyor. Group'lardan birisine sadece Print Permission'ı Allow, diğerine ise Full Allow Permission verebiliriz. Yani duruma göre Group'lara izinler ya da kısıtlamaları bu yolla getirebiliriz.
Printer'ı kuranların da Permission'ları da Full Allow olur.

Printer'lar belli zamanlarda kullanıma kapatılabilirler. Bunun için Schedule programlanabilir. Bunu yapmak için aşağıdaki yol izlenir;
Print Management > Printer > Right Click > Properties > Advanced > Available from __ to __
Yukarıda Printer'ın çalışacağı zaman aralığını belirleriz.
Örneğin mesai saatlerinin dışında Printer'ların kullanımını kapatarak gereksiz kullanıma karşı bu sayede önlem almış olacağız.

Yukarıdaki şekilde oluşturulmuş bir zaman kısıtından bir Admin olarak etkilenmek istemeyebiliriz. Admin olmak bu kısıta takılmanın doğrudan önüne geçmek için yeterli değildir. Bu durumda yapmamız gerekenler şunlardır;
Zaman kısıtı oluşturduğumuz Printer Device'ın Port'undan bir Printer daha Install ederiz. Yeni ürettiğimiz Printer'ın Permission'larından Everyone'ı kaldırırız ve Admin'i Full yetkili yaparız. Full yetkimiz olduktan sonra o Printer'dan Schedule'ı kaldırabiliriz veya zaten yeni bir Printer olduğu için o zaman kısıtı bu Printer'da görünmeyebilir bile. Artık o Printer Device'ı kullanmanın iki farklı yolu oluşur doğal olarak. Client'ten bir çıktı almak istediğimizde zaman kısıtlı Printer Device'ı yeni ürettiğimiz Printer üzerinden çekerek baskı alabiliriz. Hatta bu Printer'ın Hidden Share ederek diğer kullanılar için görünmez halde paylaşabiliriz. Özetle Ctrl + P yaptığımızda ürettiğimiz yeni Printer'ı seçeriz.

Network Printer'ı File Server'a Connect etmek

Network Printer Device'ın farkı fiziksel olarak üzerinde Ethernet Port'u bulundurması yani doğal olarak bir IP Adresine sahip olmasıdır.
Print Server > Print Manager > Printers > Add Printer > Add a TCP/IP > Type of Device: TCP/IP -> Printer Device'ın IP'si girilir. > |+|Auto
Cihaz Network'e bağlı ise Connect olur. Heniz cihaz bağlı değilse; listeden cihazın markası ya da Generic seçilir. Generic için üzerinde genel bir kart olan cihaz diyebiliriz. Bu noktadan sonra Printer Install Wizard başlar.

Network Printer Device'lara yazıcının kendi monitöründen de IP atanabilir. Eğer kendi monitörü yok ise bir cihaza bağlayıp aynı modem arayüzüne bağlanır gibi IP konfigürasyonu tarayıcı üzerinden de yapılabilir.

DHCP Server(Dynamic Host Configuration Protocol)

DHCP, otomatik IP almaya ayarlanmış makinelere IP konfigürasyonu yapmayı sağlayan servistir. DHCP Microsoft'a özel değildir. Fakat kurs için Microsoft DHCP öğretilecek. Alternatifi olan Firewall ya da Router'ın üzerindeki DHCP'leri de kullanabilir olacağız. Çalışma prensip ve mantıkları aynıdır. Hatta Microsoft DHCP en kapsamlı olanıdır.

DHCP Static IP atamaya göre çok avantajlıdır. IP çakışmasını önler, herhangi bir yere verdiğimiz IP'leri not etmek zorunda kalmayız vs.

Client'in DHCP Server'dan IP Kiralama Süreci

Otomatik IP almaya ayarlanmış son kullanıcı cihazına "DHCP Client" denir.
Bu cihaz bir IP alabilmek için bulunduğu Network'e Broadcast yapmak zorundadır.
IP'si olmayan bir makinenin IP Adresi 0.0.0.0, MAC Adresi ise sadece o makineye özel olan fiziksel ID'sidir. Şuanda bunun ne olduğunu bilmiyoruz fakat makine MAC Adresi bilgisine bunu yazar.
Yapacağı Broadcast'te hedefin(Destination) IP Adresi 255.255.255.255(Broadcast ID)'dir, MAC Adresi ise FF.FF.FF.FF'dir.

  1. Aşama DHCP Discover
  2. Aşama DHCP Offer
  3. Aşama DHCP Request
  4. DHCP Acknowlegdement

DHCP Discover

IP Adresisi olmayan cihazın(Client'in), IP Adresi alabilmek için yaptığı Broadcast'tir. Amaç ortamdaki DHCP Server'ı bulmaktır. Yapılan bu Broadcast maksimum Router'a kadardır. Router'lar Broadcast yayınları geçirmezler, Client o Router'a kadar olan yola Network denir.

DHCP Offer

DHCP Offer, DHCP Server'ın DHCP CLient'lere önerdiği IP Adresini barındıran bir Broadcast mesajıdır. Scope'taki ilk müsait IP Adresi Scope'tan çıkartılır ve DHCP Server'a Broadcast paketinin içerisine yerleştirilir. Bu şekilde Client'lere bu IP Adresini önerir. Broadcast'ler aynı anda gönderildikleri için önce hangi paket Client'e ulaşırsa, Client o önerilen IP Adresini kullanılır.

DHCP Request

DHCP Client, DHCP Server'dan gelen Broadcast'e göre DHCP Client'in hangi IP Adresini kabul ettiğini(istediğini) bildiren bir Broadcast paket gönderdiği aşamadır. Bu şekilde Network'teki herkes ve DHCP Server, DHCP Client'in DHCP Server'da oluşturulan Scope'tan hangi IP Adresini aldığını öğrenmiş olur.

DHCP Acknowlegdement(DHCP AK)

Bu aşamada DHCP Server, DHCP Client'ten gelen Request'i kabul eder. Kabul ettiğini Broadcast ile yayınlar.
Bu aşama ilk bakışta biraz gereksiz gibi görünebilir fakat şöyle bir fonksiyonu varıdr; Örneğin Network'te birden fazla DHCP Server bulunuyor, DHCP1 ve DHCP2 kendi Scope'larından birer IP önerdiler. Diyelimki DHCP1'in önerdiği IP önerildi. Bu durumda DHCP2 aralarındaki yarışı kaybetmiş olur. DHCP2 kaybettiği içinde NAK adıverilen bir Broadcast yayınlar. Bu yayının fonksiyonu şudur; Eğer DHCP Server Scope'tan bir IP çıkarırsa normal şartlarda bunu yerine koyamaz fakat NAK Broadcast'i bunu sağlar. Daha doğrusu Network'un bu IP'nin tekrar Scope'ta olduğunu ve tekrar önerilebileceğini bildirir.

Yukarda bahsettiğimiz bu 4 aşamaya kısaca DORA ismi verilmiştir.
DHCP Client, her kapatılıp tekrar açıldığında süreç Discover aşamasından başlatılar. Fakat bu defa kendisine daha önceden verilen IP Adresinin ona tekrardan verilmesini talep eder. Aynı IP Adresini geri verilebilmesi için Client'ler tarafından bu IP Adresleri kiralanırlar. Bu kiralama süreci gerçektende aslında bir kontrat'tır. Microsoft DHCP için bu IP Adreslerinin kiralanabilme süresi 8 gündür. Bunun anlamı şudur; Bir IP Adresi DHCP Server'ın Scope'undan çıkarıldıysa ve bir DHCP Client'e önerilip o Client tarafından kiralanıldıysa, o Client çalışsın veya çalışmasın her koşulda 8 gün boyun o IP Adresini meşgul eder. Dolayısıyla o IP Adresi Client kapalıyken başka bir Client'e atanmaz. İstenirse bunun engellendiği yani Client'in kapandığı durumda onun kapanmadan önce sahip olduğu IP Adresini direkt olarak başka bir DHCP Client'e önermesi de mümkündür. Bu durum genellikle ISP'lerin müşterilerinin Modem'lerinin dış bacaklarına atadıkları gerçek IP'ler için gerçekleşir. Gerçek IP'ler çok kıymetli ve sınırlı oldukları için sürekli en verimli şekilde kulanılmak durumundadırlar. Bu sebeple kendimiz kurduğumuz Private IP'ler kullandığımız Network'ler için IP'lerin kiralanması, hatta süresiz kiralanması(Süresiz kiralama da mümkündür) en mantıklısıdır.

IP Renewing(IP Yenilemek)

    Bir Client'in sahip olduğu IP'yi bırakmasının üç yolu vardır;
  1. Makine Restart edilirse IP yenileme başlar.
  2. Ethernet Kartı Restart edilirse IP yenileme başlar.
  3. cmd > ipconfig /release komutu ile IP bırakılır.

Komut ile IP bıraktıktan sonra tekrar Discover yapabilmek için;
cmd > ipconfig /renew
Yukarıdaki komut ile Client'in IP'si yenilenir. Network'te başka bir DHCP Server var ise veya IP'nin kiralandığı günün üzerinden 8 gün geçtiyse bu durumda bırakılan IP'nin geri alınacağının garantisi yoktur.

DHCP Client'i 8 gün boyunca kapatmazsak ve DHCP Client, DHCP Server'a ulaşamazsa neler olur?

DHCP Client'i hiç kapatmadığımızı düşünelim, Server ve CLient arasında toplam kiralama süresinin %50'si dolduğunda Client Server'a gidip 3.aşamadan yani DHCP Request'den başlar. Özetle DHCP Client ve DHCP Server arasında tekrar sözleşme yapılır ve yeni bir 8 günlük sözleşme daha yapılır.
DHCP Client sözleşmedeki sürenin %50'di dolduğunda, DHCP Server'a Request Broadcast atar. Eğer bu noktada DHCP Client, DHCP Server'a ulaşamazsa, sözleşmedeki sürenin %87.5'i dolduğunda tekrar Request Broadcast yapılır. Tekrar DHCP Server'a ulaşılamazsa sözleşmedeki sürenin %100'ü tamamlandığında tekrar Request Broadcast yapılır. Eğer bu noktada iletişim kurulabilirse Request yapılır ve yeniden sözleşme yapılıp aynı IP Adresi kullanılmaya devam eder.
Eğer %100 dolduğunda DHCP Server'a ulaşılamıyorsa bu durumda DHCP Client IP'yi bırakır. Doğal olarak sürenin tamamı dolana kadar DHCP Client üzerindeki IP'yi kullanmaya devam etmiştir. DHCP Client artık Client olur yani otomatik olarak bir DHCP Server'dan IP alamadığı ve halihazırda üzerinde olan IP'yi bıraktığı için artık IP Adresi 0.0.0.0'a geri döner. Bu noktada Client DHCP Discover Broadcast yapar.Eğer o Network'te başka bir DHCP Server var ise süreç en başından başlar ve sonuç olarak DHCP Server'ın Scope'undan bir IP kiralayıp hayatına o IP Adresi ile devam eder.
Eğer bu noktada da herhangi bir DHCP Server'a ulaşamıyorsa, hala otomatik olarak IP almaya ayarlı olduğundan dolayı İşletim Sistemi üzerinde bulunan APIPI adıverilen 169.254.0.0'tan başlayan Scope'tan bir IP alır.
APIPA TCP/IP kullanan her İşletim Sisteminde gömülüdür. İşletim Sisteminin bir marifeti değildir. Fakat o sistem içerisinde bu durumlar için bulunan bir Scope'tur. Doğal olarak APIPA, bahsettiğimiz Client'e denk her makinede vardır ve o Network'te hiç bir Client DHCP Server'a erişemezse nihayetinde APIPA'dan bir IP Adresi üstleneği anlamına gelir.
Dolayısıyla her APIPA'dan IP alan makinede IP Adresleri aynı olabilir. Bu da APIPA'dan IP alan makinelerin aynı Network'te olsalar bile iletişim kuramalarına sebep olur. Bunu önlemek için Client APIPA'dan IP atamadan önce almayı istediği IP'yi içeren bir Broadcast yapar. Bu Broadcast'i 3 defa yapar. Network'te eğer bu IP'yi alan bir Client yok ise kendisi alır. Eğer o IP'ye sahip olan bir Client var ise bu durumda bir sonraki IP için aynı süreç uygulanır.
Bir Client'in APIPA'dan IP Adresi alması aslında o Client'in çadır kurması anlamına gelir. Daha önceden DHCP Server'dan kiraladığı IP Adresini de ev olarak hayal edebiliriz. Evsiz kaldığı için çadır kurmak durumunda kalır.
Client APIPA'dan IP aldıktan sonra her 30 saniyede bir DHCP Discover Broadcast yapar. Bu da APIPA'ya düşen Client'lerin sayısına göre Network'te çok ciddi bir trafik oluşması anlamına gelir.
APIPA'daki makinelere Gateway atanmaz, doğal olarak o Network'un dışına paket gönderemezler, internet ile ilişkileri olmaz. Yani iletişimleri LAN ile sınırlıdır.

Client'in APIPA'dan bir IP Adresi aldığını fark ettiğimizde, Client'in Network kablosunu kontrol ederiz. Kabloda bir sorun yok ise DHCP Server'la iletişimde bir sıkıntı olduğu anlamına gelir. Sadece iletişim de değil direkt olarak DHCP Server'da bir sorun olduğu anlamına da gelebilir.
Printer Device'lar da APIPA'ya düşebilirler, bu durumda APIPA'ya düşen Printer'a bağlı bir Client'lerden çıktı alınmak istenirse alınamaz. Çünkü Active Directory üzerinden bir Ticket ile iletişim sağlanır.
Domain Client'lerinden Priter Device'a paylaşım gitmeden önce Printer Server'a Client Ticket ile yönlendirilir. Printer Server gelen Client'in Ticket'ının doğruluğunun kontrolü için Domain Controller ile iletişime geçer. Doğru ise baskıyı yapmak için Device'a paketi gönderir, değilse baskı yapılmaz. Bu işlem Active Directory üzerinden gerçekleşir.
